## 修復實作（全在 web repo 內）

### 修復 #1：硬編碼事實表注入（`workflows/generate-report/steps.ts`）
在 `aiGenerateR` 的 userPrompt 開頭注入：

```
════════════════════════════════════════════
【合盤事實表（違反即不合格）】
════════════════════════════════════════════
李進壹：八字「甲戌 丙子 丙申 乙未」
  → 日主：丙  日支：申
  → 年支：戌  生肖：狗
馮思齊：八字「丙子 丁酉 乙亥 丁亥」
  → 日主：乙  日支：亥
  → 年支：子  生肖：鼠

【雙方年支生肖關係】戌(狗) × 子(鼠) = 中性（無合無沖無刑無害）
【雙方日支關係】申 × 亥 = 六害
【雙方日干關係】丙 × 乙 = 乙(木)被生 丙(火)
【十神（以李進壹為我）】丙見乙 = 正印
【十神（以馮思齊為我）】乙見丙 = 傷官

※ 所有事實來自 lunar-python 精算，禁止 AI 自行推測。
════════════════════════════════════════════
```

核心函式（全部硬編碼查表，不靠 AI 推理）：
- `parseBazi()` — 從八字字串抽取年支/日干/日支
- `ZHI_TO_ZODIAC` — 地支 → 生肖
- `dizhiRelation()` — 六合/六沖/六害/三合/自刑/子卯刑 全查表
- `tianganRelation()` — 五合/四沖/生剋 全查表
- `shishen()` — 十神計算（陰陽 + 五行生剋）

### 修復 #2：Prompt 硬規則（`prompts/r_plan_v2.ts`）
加入「命理正確性硬規則」6 大條：

1. 生肖以年柱為唯一依據，查表對應
2. 地支合/沖/刑/害必須查表，禁止發明（明列所有關係）
3. 天干合/沖/剋必須查表（明列甲己/乙庚/丙辛/丁壬/戊癸合；甲庚/乙辛/壬丙/癸丁沖）
4. 十神關係必須查表（同陰陽+五行生剋判定）
5. 排盤數據為準，禁止發明
6. 五行強弱論述要自洽（禁止「完全沒 X」+「地支藏干補 X」同時出現）

### 修復 #3：Post-generation QA 強化（`validateReportAgainstData`）
- R 方案針對當前成員姓名，搜尋「{姓名}...屬X」模式
- 若生肖錯 → 自動替換為正確生肖
- 新增「禁忌關係詞」檢查（子戌相刑/丙庚相沖/狗鼠相害等），發現記錄

### 修復 #4：Quality Gate 硬閘門（`qualityGate`）

- 發現「子戌相刑」「丙庚相沖」「狗鼠相害」「狗兔相沖」「馬雞相沖」等不存在關係 → `hardFailure`
- 觸發上游 workflow 重新生成（最多 3 次）

### 修復 #5：UI 雙人命盤事實卡（`app/report/[token]/page.tsx`）

- 雙人命盤對照卡（avatar + 生肖 + 日主 + 八字 + 雙方生肖關係）

## 剩餘已知風險（5% 空間）

1. **五行論述推理**（95分）：AI 可能在複雜論述中自相矛盾，prompt 規則依賴 AI 邏輯自洽能力而非硬閘門。建議未來加「邏輯一致性檢查」腳本。

2. **命宮/紫微主星**：目前只硬編碼八字、生肖、十神。紫微命宮主星的核對仍依賴 AI 對 `analyses.details` 的解讀。建議 Python 排盤 API 把紫微主星也放 `client_data` 給 AI 直接引用。

3. **西洋占星相位**：太陽/月亮/金星位置的合盤互動，目前 prompt 沒做硬編碼。準確度視 `西洋占星` analysis 的 `details` 完整度而定。

4. **Round 1 的歷史 4 份報告**：已出貨給客戶的報告不會自動重生，需人工決定是否重發或補償。建議：對 R1/R3/R4 客戶發致歉信+免費重生。
