## ③ 真因(R2 失分主因、Apple HIG 視角)

### 🔴 P0 — 行寬 max-width 規則未進 production DOM(本應 +7 到 90+)
- **CSS 寫了**:`page.tsx:1634` `.report-p > p, .report-p > ul, .report-p > ol, .report-p > blockquote { max-width: 800px; margin-bottom: 1.75rem; }`
- **實測 DOM**:`getComputedStyle('.report-p > p').maxWidth = "none"` × 8 個樣本全部
- **rendered width**:1299px(超 Apple HIG 65 字 ≈ 720-820px 上限 60%)
- **疑因**:`<p>` 不是 `.report-p` 的「直接子元素」(可能被 `<div class="report-p">` 包再包 `<div>` 才到 `<p>`、或 dangerouslySetInnerHTML 渲染後 DOM 結構意外)
- **修法**:把 selector 改 `.report-p p, .report-p ul, .report-p ol, .report-p blockquote`(去 `>`、descendant 通用)、預期 +7 分達 93/100
- **若 P0 修了**:R2 → **93 / 100**(差 2 分達 95+ 鐵律)

### 🟡 P1 — `.report-p` 自身 margin-bottom 48px 與 `.report-p > p` 28px 段距不一致
- **CSS L1632**:`.report-p { margin-bottom: 3rem(=48px) }` ← 套到 div.report-p 整塊
- **CSS L1634**:`.report-p > p { margin-bottom: 1.75rem(=28px) }` ← 套到內 p
- **問題**:48px 和 28px 不一致、Bringhurst 「段距 = 行距 × 1.5」(行距 32.4px × 1.5 = 48.6 ≈ 48px、48 對、28 不對)
- **修法**:統一 `.report-p > p margin-bottom: 1.75rem → 2rem`(28→32、8pt grid +1 分)、或全改 48px
- **若修**:8pt grid 8 → 10、+2 分

### 🟢 P2 — 觸控 / 邊距響應未動(R1 已列、未補)
- step-badge 28×28 → 32×32 / 44×44(+2 分)
- desktop wrapper `lg:px-8`(+3 分)
- 共 +5 分
